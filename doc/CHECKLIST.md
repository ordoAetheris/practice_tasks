# Чек-лист самопроверки решения

Пройди этот чек-лист **после** того, как решение готово и тесты проходят.
Подходит для любой задачи из проекта.

---

## 0. Базовое

- [ ] Код компилируется без ошибок (`mvn compile`)
- [ ] Все тесты зелёные (`mvn test -Dtest="com.practice.taskXX_name.*"`)
- [ ] Решение лежит в пакете `solution/` (не коммитится в репозиторий)

---

## 1. Декомпозиция (Single Responsibility)

- [ ] Модели данных (record/class) — в отдельных файлах, а не inner-классы в сервисе
- [ ] Хранение данных (репозиторий/хранилище) отделено от бизнес-логики
- [ ] Каждый класс можно описать одним предложением без слова "и"
- [ ] Нет God-класса, который одновременно хранит, ищет, валидирует и трансформирует

---

## 2. Абстракции и зависимости (DIP, ISP)

- [ ] Сервис принимает зависимости через конструктор (constructor injection)
- [ ] Сервис зависит от интерфейса, а не от конкретного класса репозитория
- [ ] Интерфейсы содержат только нужные методы (нет лишних "на будущее")
- [ ] Можно подменить реализацию репозитория без изменения сервиса

---

## 3. Контракты и API

- [ ] Публичные сигнатуры методов из задания **не изменены** (типы параметров, возвращаемый тип)
- [ ] Исключения бросаются тех типов, что указаны в javadoc (`IllegalArgumentException`, `IllegalStateException` и т.д.)
- [ ] Сообщения в исключениях понятны (описывают что пошло не так)
- [ ] Новые публичные методы (если добавлены) имеют осмысленный контракт

---

## 4. Структуры данных

- [ ] Для поиска по ключу используется `Map`, а не перебор `List`
- [ ] Для уникальности — `Set`, а не ручная проверка `contains` + `add`
- [ ] Нет лишнего копирования коллекций (`.stream().collect()` там, где хватит `.toList()`)
- [ ] Выбор между `HashMap`/`TreeMap`/`LinkedHashMap` — осознанный

---

## 5. Чистота кода

- [ ] Классы, методы и переменные названы по конвенции Java (camelCase, PascalCase)
- [ ] Имена отражают суть, а не тип (`users` вместо `userMap`, `findByAuthor` вместо `method1`)
- [ ] Нет дублирования логики (одинаковый код в двух местах → вынести в метод)
- [ ] Нет закомментированного кода, мёртвых методов, оставшихся `// TODO`
- [ ] Форматирование единообразное (отступы, скобки, пустые строки)

---

## 6. Тесты

- [ ] Покрыты все сценарии из `test_hints.md` → раздел "Обязательно протестировать"
- [ ] Есть тесты на ошибочные сценарии (`assertThatThrownBy(...)`)
- [ ] Есть хотя бы один граничный случай (пустые данные, ноль, null)
- [ ] Каждый тест проверяет **одну** вещь (один assert или группу связанных)
- [ ] Названия тестов читаются как спецификация: `shouldThrowWhenNoCopiesAvailable`
- [ ] Тесты не зависят от порядка выполнения (нет общего мутабельного состояния)

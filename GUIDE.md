# Practical Java — Руководство

## Что это

100 практических задач для подготовки к Java-собеседованиям в российских компаниях.
Формат взят с реальных собесов (Sber, X5, Tinkoff, VK, Яндекс и др.).

Каждая задача — это класс-заготовка с описанием, сигнатурами методов и `//TODO implement`.
Твоя работа: реализовать логику, вынести модели в отдельные классы/интерфейсы если нужно,
написать тесты, убедиться что всё работает.

## Рекомендуемый процесс

Эти задачи имитируют тестовое задание на собеседовании. Лучший способ тренировки —
относиться к ним именно так:

### Зачем таймер

Умение писать код — это одно. Умение решать задачу за ограниченное время — другое.
На практике второе важнее первого.

На собеседовании у тебя час. Не два, не "пока не доделаю". Если за час ты написал
красивую архитектуру, но не работающий код — ты не прошёл. Если написал грубый,
но работающий и протестированный код — прошёл. Интервьюер оценивает результат
в рамках ограничения, а не потенциал при бесконечном времени.

На работе то же самое, только масштаб другой. Задача в спринте — это таймер
на 2 недели. Инцидент в проде — таймер на минуты. Ревью перед релизом — таймер
до конца дня. Везде есть дедлайн, и навык укладываться — это не про скорость
печати, а про умение мгновенно отсекать лишнее: не тот паттерн, не тот уровень
абстракции, не те приоритеты. Именно это тренируется таймером.

Без таймера мозг работает в режиме "а давай ещё улучшим". С таймером — в режиме
"что минимально необходимо чтобы это работало правильно". Второй режим и есть
инженерный подход.

### Поставь таймер

| Уровень | Целевое время | Hard cap (+15%) |
|---------|---------------|-----------------|
| Базовые (1-30) | 60 мин | 69 мин |
| Средние (31-60) | 75 мин | 86 мин |
| Продвинутые (61-80) | 90 мин | 103 мин |
| Хардкор (81-100) | 120 мин | 138 мин |

Целевое время — реализация + тесты. Таймер не выключается после звонка.

**Три зоны:**
- **В пределах целевого** — задача решена. Чисто.
- **Целевое → hard cap (+15%)** — задача решена, но ты в минусе. На собесе
  это уже нервный взгляд интервьюера на часы. На работе — задача, которая
  не влезла в спринт. В проде — лишние минуты даунтайма. Формально зачёт,
  но ты знаешь что надо быстрее.
- **После hard cap** — не решена. Точка. Неважно, сколько осталось дописать.
  На собесе тебе скажут "спасибо, мы вам перезвоним". В спринте — перенос
  задачи, разговор с лидом, пересмотр оценки.

**Режимы работы:**

**Разведка** (первый подход) — без таймера. Цель: разобраться в задаче,
набить шишки, понять подвохи. Потратил час на ненужный репозиторий — отлично,
в следующий раз не потратишь. Разведка не считается попыткой и не считается
решением. Это тренировка, а не матч.

**На время** (второй+ подход) — `git checkout`, чистый файл, таймер, погнал.
Вот здесь считается. Не уложился в hard cap — стёр, разобрал почему,
попробовал снова. С каждой попыткой быстрее: первая попытка обычно уходит
на осознание что ты делаешь лишнее, вторая — на правильную архитектуру,
третья — чистая уверенная сдача.

**Совет:** не тренируйся уставшим на сложных задачах — возьми задачу проще.
Растягивать таймер — неправильный навык. Лучше чисто решить базовую за 40 минут,
чем мучительно ковырять продвинутую за 3 часа.

**Задача со звёздочкой:** по ходу решения оставляй короткие заметки прямо в коде —
что ты считаешь тонким местом, что нерасширяемо, где сознательно срезал угол:

```java
// NOTE: линейный поиск по всем займам — на проде был бы индекс по bookId
// TODO: если добавить историю изменений — придётся менять структуру хранения
// NOTE: не потокобезопасно, достаточно для in-memory без конкурентного доступа
```

Это демонстрирует инженерную зрелость: ты принял решение осознанно, а не потому
что не знаешь лучшего способа. На реальном собесе такой комментарий — это плюс:
интервьюер видит, что ты понимаешь ограничения своего решения. На ревью — тоже:
коллега сразу видит, что это не баг, а trade-off.

Сюда же можно вписывать свои архитектурные и паттерновые наблюдения:

```java
// NOTE: здесь напрашивается Strategy для разных политик поиска,
//       но для одной реализации это оверинжиниринг
// NOTE: в проде это был бы JOIN по bookId+userId, а не два отдельных lookup
// NOTE: нарушаю OCP — при добавлении нового статуса придётся менять switch.
//       Для in-memory допустимо, на проде — вынес бы в enum с поведением
```

При ревью через нейросеть эти заметки оцениваются отдельным этапом (см. промпт
в `REVIEW_PROMPT.md`): сначала основная оценка кода, потом — вменяемость и
архитектурная ценность заметок. Это показывает глубину понимания, не влияя
на оценку самого решения.

Но: заметки пишутся **по ходу**, на автомате, за секунды — не вместо кода.
Если заметка длиннее одной строки — ты отвлёкся.

### Workflow

```bash
TASK=task01_library_service

# 1. Прочитай задание
cat src/main/java/com/practice/$TASK/*.java

# 2. Запусти таймер
# 3. Реализуй (прямо в файле задачи — заполни TODO)
# 4. Вынеси модели в solution/ если нужно (gitignored)
# 5. Напиши тесты
# 6. Проверь
mvn test -Dtest="com.practice.$TASK.*"

# Если время вышло или хочешь начать заново:
git checkout -- src/main/java/com/practice/$TASK/
rm -rf src/main/java/com/practice/$TASK/solution/
git checkout -- src/test/java/com/practice/$TASK/
```

Реализация пишется **прямо в файле задачи** — заполняешь тела методов.
Если нужно вынести модели (records/enums) в отдельные файлы — клади их в `solution/`
внутри пакета задачи. Эта папка в `.gitignore`, задание остаётся чистым.

`git checkout` сбрасывает файл к исходному состоянию с TODO — можно начать заново.

## Как работать с задачей

### 1. Прочитай задание
Открой класс задачи (например `LibraryService.java`). В javadoc — полное описание:
что реализовать, какие сущности, какие правила и ограничения.

### 2. Реализуй
- Методы помечены `//TODO implement` — это твоя работа
- Inner records/interfaces/enums — заготовки моделей. Можно оставить внутри
  или вынести в отдельные файлы в `solution/`
- Выбор структур данных, обработка ошибок — всё на тебе

**ВАЖНО — типичная ловушка:**

Эти задачи — **in-memory сервисы**, а не микросервисы с базой данных.
Реализация пишется **внутри выданного класса**. Хранилище — это `Map`/`List`
прямо в полях этого класса. Ты реализуешь тела методов и добавляешь приватные
хелперы по необходимости.

Чего делать **НЕ нужно**:
- Создавать отдельные Repository/DAO классы — это обёртка над Map, что бессмысленно
- Создавать отдельные Service-классы для каждой сущности — это создаёт ненужные
  зависимости между сервисами и усложняет код
- Добавлять Mockito/моки — если нет внешних зависимостей, мокать нечего
- Делать constructor injection интерфейсов — нет внешних зависимостей для инжекта
- Добавлять ReentrantLock/AtomicLong — если в задании не сказано про многопоточность,
  не нужно
- Менять публичные сигнатуры методов (типы параметров, возвращаемый тип)

**Вынести в отдельные файлы** можно и нужно только **модели данных** (records/enums),
если они загромождают основной класс. Логика остаётся в сервисе.

Думай об этом как о тестовом задании на собеседовании: у тебя час, и задача —
показать чистый рабочий код, а не архитектуру enterprise-приложения.

### 3. Напиши тесты
- Тестовый класс уже создан, но пустой
- В `test_hints.md` — подсказки что нужно обязательно проверить
  и какие продвинутые кейсы стоит рассмотреть
- Подсказки говорят ЧТО тестировать, не КАК — это тоже часть навыка

### 4. Проверь
```bash
# Запуск тестов конкретной задачи
mvn test -Dtest="com.practice.task01_library_service.*"

# Запуск main-метода (smoke test)
javac src/main/java/com/practice/task01_library_service/LibraryService.java
java -cp src/main/java com.practice.task01_library_service.LibraryService

# Все тесты
mvn test
```

## Инструкция для AI-ассистента (Claude, GPT, etc.)

Этот проект — учебный. Ученик решает задачи самостоятельно.

### Роль ассистента

**Помогай разобраться, но не решай за ученика.**

### Что можно делать
- Объяснить концепцию (что такое LRU, как работает Trie, зачем нужен fencing token)
- Подсказать направление ("подумай какая структура данных даёт O(1) на get и put")
- Помочь разобраться с ошибкой компиляции или упавшим тестом
- Объяснить почему тест должен быть именно таким
- Ревьюить готовое решение: указать на проблемы, предложить улучшения
- Помочь с синтаксисом JUnit/AssertJ если ученик не знаком

### Чего нельзя делать
- Писать реализацию методов (даже "для примера")
- Писать готовые тесты
- Давать готовый код который можно скопировать
- Решать задачу "по шагам" диктуя каждую строку

### Типичная ловушка — оверинжиниринг

Если ученик начинает создавать отдельные Repository/DAO-классы, отдельные Service
для каждой сущности, интерфейсы, constructor injection, Mockito — **останови его**.

Это in-memory задачи. Правильный подход:
- Данные хранятся в Map/List полях того класса, который выдан
- Реализация — в телах методов этого класса + приватные хелперы
- Модели (records) можно вынести в отдельные файлы, но это необязательно
- Тесты создают сервис через `new Service()` и проверяют напрямую
- Mockito не нужен — мокать нечего

Создание Repository как обёртки над Map — бессмысленная абстракция.
Создание отдельных сервисов для каждой сущности создаёт ненужные зависимости.
Задача имитирует тестовое на собес: час на решение, не enterprise-архитектуру.

### Формат помощи — примеры

**Ученик:** "Не понимаю как сделать LRU кэш"
**Плохо:** "Вот реализация: class LRUCache { ... }"
**Хорошо:** "LRU вытесняет элемент, к которому дольше всего не обращались.
Подумай: какая структура позволяет быстро находить элемент по ключу
И какая — быстро перемещать элемент в конец при обращении?
Тебе нужна комбинация двух структур данных."

**Ученик:** "Тест падает, не понимаю почему"
**Хорошо:** Попросить показать тест и реализацию, указать на конкретное расхождение
между ожидаемым и фактическим поведением.

**Ученик:** "Какие тесты писать?"
**Хорошо:** "Посмотри test_hints.md рядом с тестовым классом — там есть подсказки.
Начни с обязательных кейсов."

### Ревью кода

Когда ученик показывает готовое решение, проверь:
1. **Адекватность архитектуры** — решение соразмерно задаче? Нет оверинжиниринга?
   Если ученик создал Repository/DAO для обёртки над Map — укажи что это лишнее.
   Если создал отдельные Service для каждой сущности — объясни почему это проблема.
2. Соответствие требованиям из javadoc задания
3. Корректность бизнес-правил
4. Выбор структур данных (эффективность)
5. Качество тестов (покрытие, осмысленность, отсутствие моков без необходимости)
6. Чистота кода (именование, дублирование, сложность)

Давай конкретную обратную связь, не абстрактную.
"Метод search() делает полный перебор за O(n*m), можно ли лучше?" —
а не "код можно улучшить".

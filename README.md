# Practical Java

100 практических задач для подготовки к Java-собеседованиям в российских компаниях.

Формат взят с реальных собесов: Sber, X5, Tinkoff, VK, Яндекс и др.

## Быстрый старт

```bash
git clone <repo-url>
cd practical_java

# Открой задачу — прочитай javadoc
cat src/main/java/com/practice/task01_library_service/LibraryService.java

# Реализуй в пакете solution/
# Запусти тесты
mvn test -Dtest="com.practice.task01_library_service.*"
```

## Как работать

1. **Поставь таймер** — базовые 60 мин, средние 75, продвинутые 90 (hard cap +15%)
2. **Прочитай задание** — javadoc в файле задачи описывает что реализовать
3. **Реализуй** — заполни TODO прямо в файле, модели можно вынести в `solution/`
4. **Напиши тесты** — тестовый класс создан, подсказки в `test_hints.md`
5. **Проверь** — `mvn test`
6. **Оцени себя** — чек-лист в `doc/CHECKLIST.md`, ревью через `doc/REVIEW_PROMPT.md`

Первый подход — разведка (без таймера, не считается). Со второго — на время.
После hard cap не решена, неважно сколько осталось. `git checkout` — заново.

> **Важно:** это in-memory задачи. Реализация — внутри выданного класса.
> Хранилище — `Map`/`List` в полях. Не нужно создавать Repository, отдельные
> Service-классы, интерфейсы или подключать Mockito. Подробнее — в [GUIDE.md](GUIDE.md).

## Философия задач: инженерный подход, а не паттерны ради паттернов

Если ты крепкий разработчик — эти задачи могут показаться обманчиво простыми.
Соблазн будет сделать "как на проде": выделить репозитории, создать интерфейсы,
прокинуть зависимости через конструктор, написать моки. **Это ловушка.**

Эти задачи тренируют не знание паттернов, а **инженерную зрелость** — умение
отличить ситуацию, где паттерн нужен, от ситуации, где он вреден. На реальном
собеседовании оверинжиниринг in-memory задачи за час — это минус, а не плюс.
Интервьюер видит, что кандидат не оценил скоуп, не уложился во время и не решил
задачу, потому что строил архитектуру которую никто не просил.

### Что здесь отрабатывается

**YAGNI (You Aren't Gonna Need It)** — реализуй только то, что требуется.
Repository для обёртки над HashMap — это код, который не решает никакой задачи.
Интерфейс для единственной реализации — мёртвая абстракция. Если через месяц
понадобится подменить хранилище — тогда и выделишь интерфейс. Не раньше.

**SOLID — правильно понятый, а не карго-культовый:**
- **SRP** — один класс, одна ответственность. Но ответственность LibraryService —
  это управление библиотекой целиком, а не "быть прослойкой между контроллером
  и репозиторием". Map внутри сервиса — это деталь реализации, не нарушение SRP.
- **OCP** — код открыт для расширения. Но расширение должно быть реальным,
  а не воображаемым. "А вдруг мы заменим Map на базу?" — вот когда заменишь.
- **DIP** — зависимость от абстракций. Но если абстракция одна и её подмена
  не предвидится — это не инверсия зависимостей, а лишний слой.

**DRY** — не повторяйся. Но три похожих строки — это не повторение, а
три бизнес-правила. Преждевременная абстракция хуже дублирования.

**GRASP** — всё это не противоречит GRASP. Information Expert говорит:
ответственность должна быть у того, кто владеет данными. Сервис владеет
коллекциями — он и работает с ними. Low Coupling говорит: минимум
зависимостей — а ты создаёшь их на пустом месте, вынося в отдельные классы.

### Как оценить своё решение

Задай себе вопрос: **если бы это было тестовое на собеседование с лимитом
в 1 час — я бы успел?** Если ответ "нет, потому что я 30 минут строил
архитектуру" — решение переусложнено. Чистый код, работающие тесты,
соблюдение контракта — вот что оценивается. Не количество классов.
